# adnc-skill-ddl-codegen instructions

本项目的核心目标不是开发业务系统本身，而是**编写、维护、测试 skill 文件**，用于指导代理根据 DDL 生成符合约定的 Adnc Admin 模块代码。

## 项目定位

- `SKILL.md`：主 skill 定义，说明何时触发、输入要求、工作流、输出范围和约束。
- `assets\templates.md`：生成代码时使用的模板骨架。
- `references\`：参考快照与规则文档，用于对照真实项目约定，而不是直接作为产出目录；DDL 需要由用户手动提供或选择，实际生成代码应输出到 `.\src`。

## 主要工作内容

1. 编写和迭代 `SKILL.md`，让 skill 的触发条件、输入要求和输出合同更清晰。
2. 维护 `assets\templates.md` 与 `references\` 中的说明，确保生成结果与目标仓库约定一致。
3. 测试 skill 行为，重点验证：
   - 是否会先读取必要参考文件
   - 是否能正确区分普通业务表与关系表
   - 是否能生成完整文件集合
   - 是否会避免根据不完整 SQL 或 seed data 进行猜测
   - 生成代码后是否不会执行 `git add`，并只返回 `生成成功`
   - 是否会按“快速路径”直接生成普通业务表 CRUD，而不是重复确认或重复读取参考文件
4. 持续收敛提示词和规则，减少命名错误、命名空间错误、DTO 字段遗漏、校验规则缺失等问题。
5. 当 DDL 包含 `rowversion` 字段时，确保 skill 会在实体上保留 `RowVersion` 属性并实现 `IConcurrency`。

## 修改本项目时的原则

- 优先修改 skill 说明和模板，避免把仓库变成业务代码仓库。
- 所有规则都应尽量写得明确、可执行、可验证，避免模糊描述。
- 如果新增约束，最好同时补充到 `SKILL.md`、模板或参考文档中对应的位置。
- 测试 skill 时，应使用用户手动提供或明确选择的真实/接近真实 DDL 输入，而不是依赖仓库内 fallback SQL 或推断。
- 为了提升生成速度，优先把高频决策固化成默认规则，而不是把判断留到生成当下。

## 验收标准

- instruction、skill、template、reference 之间没有明显冲突。
- skill 能明确指导代理“先读什么、生成什么、不要做什么”。
- 文档更新后，新的使用者可以快速理解：这个仓库主要是**编写和测试 skill 文件**。
